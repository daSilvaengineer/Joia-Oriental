Processo Unificado Racional

Fases RUP

O Rational Unified Process (RUP) estrutura o desenvolvimento em quatro fases iterativas, cada uma com artefatos, marcos e critérios de saída bem definidos.

🎯

Visão do Projeto

E-commerce temático inspirado nas culturas da China, Japão, Índia e Turquia, com foco em joias e artesanato oriental. Entrega de uma plataforma completa com autenticação, catálogo e checkout.

👥

Stakeholders

Clientes finais (consumidores de cultura asiática), administradores da loja, desenvolvedores back-end e equipe de UX. Todos mapeados com casos de uso primários e secundários.

⚠️

Riscos Identificados

Integrações de pagamento instáveis, tempo de aprendizado do Django ORM, segurança de dados de usuário (LGPD), escalabilidade do servidor no PythonAnywhere.

📋

Artefatos da Fase

Documento de Visão, Glossário de Negócio, Lista de Riscos Inicial, Casos de Uso de Alto Nível, Estimativa de Esforço (Planning Poker).

✅

Marco de Saída — LCO

Lifecycle Objective Milestone: escopo aprovado, arquitetura de alto nível definida (Django + MTV + Service Layer), cronograma inicial criado com 3 sprints de 2 semanas.

💡

Modelo de Negócio

Venda direta B2C de joias e artesanato oriental. Receita via comissão de venda. Canal digital exclusivo com entrega nacional. Categorias: China, Japão, Índia, Turquia.

🏗️

Arquitetura Definida

Django 4.x com padrão MTV. Service Layer para lógica de negócio. Organização modular por domínio (store, ecommerce_project). Docker para conteinerização.

📐

Modelos de Dados

Entidades: Product, Category, Order, OrderItem, Customer, Review. Relacionamentos N:N entre produtos e categorias; 1:N entre pedidos e itens.

🔒

Plano de Segurança

Autenticação social (OAuth), proteção CSRF nativa do Django, validação de formulários, controle de acesso por roles (admin vs cliente), HTTPS no deploy.

🧩

Casos de Uso Refinados

UC01-Listar Produtos, UC02-Filtrar por Cultura, UC03-Adicionar ao Carrinho, UC04-Finalizar Compra, UC05-Login Social, UC06-Admin Gerenciar Catálogo.

🧪

Plano de Testes

Testes unitários com pytest-django para models e services. Testes de integração para views e URLs. Diretório /tests/ estruturado por módulo.

✅

Marco de Saída — LCA

Lifecycle Architecture Milestone: baseline da arquitetura aprovada, protótipo navegável funcionando, plano de iteração do Sprint 1 validado pela equipe.

⚙️

Sprint 1 — Core

Models principais (Product, Category), CRUD admin Django, templates base com Jinja2, rota de listagem e detalhe, configuração do Docker Compose.

🛒

Sprint 2 — E-commerce

Carrinho de compras (sessão), fluxo de checkout, integração de pagamento, pedidos e histórico, notificações por e-mail, populate_store.py para dados de teste.

🔐

Sprint 3 — Auth & UX

Autenticação social (Google/GitHub), perfil de usuário, reviews e avaliações, paginação, busca por texto, otimização de assets (CSS/JS minificado).

📁

Estrutura de Módulos

store/ (models, views, services, urls, templates), ecommerce_project/ (settings, wsgi, asgi), media/, tests/, docs/, Dockerfile, docker-compose.yml.

📊

Métricas de Qualidade

Cobertura de testes >70%, tempo de resposta <300ms em páginas de listagem, zero erros críticos (Sentry), Lighthouse score >80 para performance.

✅

Marco de Saída — IOC

Initial Operational Capability: sistema funcional em staging (PythonAnywhere), todos os casos de uso implementados, testes passando, documentação técnica gerada.

🚀

Deploy em Produção

Publicado em dasilvaengineer.pythonanywhere.com. Configurações de ambiente (DEBUG=False, ALLOWED_HOSTS, SECRET_KEY via env). Banco SQLite migrado para PostgreSQL.

🐛

Bug Fixing & Tuning

Resolução de issues de deploy (static files, media paths), ajustes de performance nas queries N+1 com select_related/prefetch_related, cache de views.

📖

Documentação Final

README detalhado no GitHub, esta documentação RUP interativa, guia de contribuição (CONTRIBUTING.md), changelog de versões, manual do administrador.

🎓

Treinamento & Handoff

Vídeo-demonstração da plataforma, guia de instalação local (Docker), tutorial de uso do painel admin, FAQ para novos desenvolvedores.

📈

Monitoramento

Logs de acesso via Django logging, alertas de erro, monitoramento de uptime. Próximos passos: integração com Sentry e métricas de conversão.

✅

Marco de Saída — PR

Product Release: sistema aceito pelo stakeholder, zero defeitos críticos abertos, documentação publicada, repositório público no GitHub com CI configurado.

Engenharia de Requisitos

Requisitos do Sistema

Requisitos funcionais e não-funcionais elicitados com técnicas de entrevista e análise de domínio, priorizados pela técnica MoSCoW.

RF-01 O sistema deve exibir um catálogo de produtos filtráveis por cultura (China, Japão, Índia, Turquia) e categoria. Must
RF-02 O sistema deve permitir que usuários adicionem produtos ao carrinho de compras e alterem quantidades. Must
RF-03 O sistema deve processar o fluxo completo de checkout: endereço, pagamento e confirmação de pedido. Must
RF-04 O sistema deve suportar autenticação social (Google/GitHub) além do login tradicional por e-mail e senha. Must
RF-05 Administradores devem poder gerenciar produtos, categorias, pedidos e usuários via painel Django Admin. Must
RF-06 O sistema deve permitir que clientes deixem avaliações e notas (1–5 estrelas) nos produtos comprados. Should
RF-07 O sistema deve enviar e-mail de confirmação após finalização de pedido com número de rastreamento. Should
RF-08 O sistema deve exibir produtos relacionados na página de detalhe com base na mesma cultura/categoria. Could
RF-09 O sistema deve possuir busca full-text por nome, descrição e tags de produtos. Should
RF-10 O sistema deve gerar relatórios de vendas por período, categoria e cliente para o administrador. Could
RNF-01 Desempenho: páginas de listagem devem carregar em menos de 500ms com paginação de 20 itens por página. Must
RNF-02 Segurança: todas as senhas devem ser armazenadas com hash bcrypt; tokens de sessão com expiração de 24h. Must
RNF-03 Disponibilidade: uptime mínimo de 99% em horário comercial; deploy zero-downtime via Docker rolling update. Must
RNF-04 Manutenibilidade: cobertura de testes automatizados ≥ 70%; toda função de service deve ter docstring. Should
RNF-05 Usabilidade: interface responsiva para dispositivos móveis (breakpoints 320px, 768px, 1200px); WCAG 2.1 AA. Must
RNF-06 Escalabilidade: arquitetura deve suportar até 500 usuários simultâneos sem degradação perceptível. Should
RNF-07 Conformidade LGPD: dados pessoais criptografados em repouso; usuário pode solicitar exclusão de conta. Must
